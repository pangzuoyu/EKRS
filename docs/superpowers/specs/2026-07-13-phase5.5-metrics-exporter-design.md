# Phase 5.5 D: `/metrics` 主端口下线 → Multi-process Sidecar Exporter

**日期**: 2026-07-13
**范围**: `routes/metrics.py` 删除、`main.py` 启动 sidecar、lifespan teardown、env vars 更新、测试改写
**非目标**: A/B/C/E,Iron Rules,audit schema,Phase 5 replay,Query/Document 数据流

---

## 目标

`/metrics` 当前随 RAG FastAPI 主进程暴露在 `:8000/metrics` 上,**完全不对外暴露在 RAG 主端口**。Prometheus 拉取改为 sidecar exporter:

- RAG 主端口 (`:8000`) 不再有 `/metrics`
- 每个 RAG pod 启动时附带 `prometheus_client` 在 `127.0.0.1:9090` 起 wsgi HTTP server
- Prometheus k8s service discovery 抓每个 pod 的 `127.0.0.1:9090`
- Multi-process 模式下 (`PROMETHEUS_MULTIPROC_DIR` 设值),Counter/Histogram 在 worker 进程内累加,在 exporter 内聚合

理由:`/metrics` Prometheus text 含内部 metric 名 (route_failures_total 等) 加 DB 路径-like label 暗示(future)。同时如果未来 RAG 加多个 worker(`uvicorn --workers N`),单进程 Counter 计数会丢;sidemode 走 multiprocess 是标准答案。

## 详细设计

### main.py lifespan 改动

```python
# imports
import os
from pathlib import Path
from prometheus_client import multiprocess, start_http_server, REGISTRY


@asynccontextmanager
async def lifespan(app: FastAPI):
    # ---- Phase 5.5 D: sidecar metrics exporter ----
    # PROMETHEUS_MULTIPROC_DIR is OPTIONAL:
    #   - Set (production multi-worker): enable multiprocess mode. Must be
    #     set BEFORE uvicorn forks worker subprocesses (set in
    #     docker-compose env or k8s Deployment env).
    #   - Unset (single worker / dev): in-memory counters; exporter serves
    #     REGISTRY directly.
    #
    # ⚠ Import-order trap (verified):
    #   prometheus_client 检查 PROMETHEUS_MULTIPROC_DIR 在 Counter
    #   construction 时(rag/ekrs_rag/observability/metrics.py:18 的 import-time
    #   side-effects)。lifespan 中读 env 已晚 — Counters 已在 master 进程用当时
    #   的 env 状态注册。Master 单进程模式运行时 Counters 是 in-memory;
    #   worker fork 后,worker 继承 master env,若 env 已设,worker 也注册 multiproc。
    #   结论:env var 必须在 `python -m ekrs_rag.main` 启动前由部署层(shell /
    #   docker-compose env / k8s Deployment env)设置;lifespan 中 os.environ.get
    #   只是为了避免 hard fail 提示。
    multiproc_dir = os.environ.get("PROMETHEUS_MULTIPROC_DIR")
    if multiproc_dir:
        p = Path(multiproc_dir)
        p.mkdir(parents=True, exist_ok=True)
        multiprocess.MultiProcessCollector(REGISTRY)
        logger.info("Multi-process metrics mode: %s", p)

    metrics_host = os.environ.get("METRICS_HOST", "127.0.0.1")
    metrics_port = int(os.environ.get("METRICS_PORT", "9090"))
    httpd = start_http_server(metrics_port, addr=metrics_host)
    logger.info("Metrics exporter listening on %s:%d", metrics_host, metrics_port)
    app.state.metrics_httpd = httpd

    # ---- existing lifespan logic (audit writer, audit index, etc.) ----
    ...

    yield

    # ---- shutdown ----
    # order matters: shutdown() halts serve_forever(); server_close() releases
    # the listening socket immediately (avoids TIME_WAIT in tests).
    httpd.shutdown()
    httpd.server_close()
    logger.info("Metrics exporter stopped")
```

### `free_port()` helper(plan 中实现)

```python
import socket

def free_port() -> int:
    """Bind ephemeral port, return its number. SO_REUSEADDR avoids TIME_WAIT bind errors."""
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
    s.bind(("127.0.0.1", 0))
    port = s.getsockname()[1]
    s.close()
    return port
```

`tests/integration/test_metrics_exporter.py` 用 `free_port()` + tmp_path,避免与并发测试撞端口。

**双端口解读**:
- 容器内 prometheus 自身 web UI / scrape API 仍用 9090,但映射到 host `19090:9090`,**不与 rag sidecar 撞**。
- prometheus 抓 `rag:9090` (在 docker compose 默认 network 里,rag service 的容器名 `rag` 解析到容器 IP),即 rag 的 sidecar exporter。**sidecar 不从 host 暴露,只有 prometheus 容器可达**。
- 单 worker 开发:`make dev` 不带 `PROMETHEUS_MULTIPROC_DIR`,sidecar 走 in-memory 模式,prometheus 也能抓到(单进程模式 scrape 一样工作)。

## 数据流

启动期:
```
lifespan 进入
    ├─ PROMETHEUS_MULTIPROC_DIR 检查 + 创建
    ├─ MultiProcessCollector(REGISTRY)  注册聚合器
    ├─ start_http_server(9090, '127.0.0.1')
    └─ httpd 句柄存 app.state.metrics_httpd
```

请求期(在 9090 端口):
```
Prometheus scraper → http GET 127.0.0.1:9090/metrics
    │
    ▼
    wsgi handler → generate_latest() → Counter/Histogram 数 (聚合 multiproc .db files)
    │
    ▼
    text/plain; version=0.0.4; charset=utf-8 body
```

关闭期:
```
lifespan 退出
    ├─ httpd.shutdown()       → serve_forever() return → 线程退出
    └─ httpd.server_close()  → 立即释放 listening socket(去 TIME_WAIT)
```

## 错误处理

| 场景 | 行为 |
|---|---|
| `PROMETHEUS_MULTIPROC_DIR` 未设 | OK,single-process 模式,REGISTRY 直接服务 |
| `PROMETHEUS_MULTIPROC_DIR` 不可写 | `mkdir(parents=True, exist_ok=True)` 失败 → RuntimeError |
| `:9090` 端口被占用 | `start_http_server` raise OSError → lifespan 失败 |
| Counter 名重复(导入冲突) | phase 5 `_isolate_prometheus_registry` session-scope 已处理 |

## 备选方案(已不采纳)

### A1: 主端口加 Bearer token 鉴权
- 优点:spec §2 推荐路径
- 缺点:仍暴露路由,且未来需要 scraper 配 token
- 决策:不采纳,user 选 "完全不对外暴露"

### A2: 周期文件 dump + json_exporter
- 优点:代码量小 (~20 行)
- 缺点:60s+ 延迟,生产不推荐
- 决策:不采纳,user 选 multi-process

### A3: Pushgateway
- 优点:实现最简单
- 缺点:Pushgateway 是 SPOF,RAG 长服务不适用
- 决策:不采纳,user 选 multi-process

### A4: 独立旁路端口(在 main app 里)
- 优点:简单
- 缺点:与 Phase 5.5 B(multi-pod)的 exporter 共存混乱
- 决策:不采纳,方案 2 在 lifespan 内启 wsgi server
